## Configuración de Inyección de Dependencias (Program.cs)

```csharp
// 1. Registrar DbContext
builder.Services.AddDbContext<HarbourDbContext>(options =>
    options.UseSqlServer(connectionString));

// 2. Registrar Repositorios
builder.Services.AddScoped<IStorageRepository, StorageRepository>();

// 3. Registrar Servicios
builder.Services.AddScoped<ILogisticsService, LogisticsService>();

// 4. Registrar Controladores
builder.Services.AddControllers();

// Resultado: Los componentes se inyectan automáticamente
// LogisticsController ? ILogisticsService ? IStorageRepository ? HarbourDbContext
```

## Validaciones Implementadas

```
Crear Item
    ?? ? ID no puede estar vacío
    ?? ? Peso propio no puede ser negativo
    ?? ? Destino (Box) no puede estar vacío
    ?? ? Tipo debe ser válido

Cargar Item en Contenedor
    ?? ? Child item existe en BD
    ?? ? Parent existe en BD
    ?? ? Parent es CompositeStorage (Pallet o Container)
    ?? ? Peso no se auto-contiene
    ?? ? Peso total no excede MaxCapacity
    ?   (current + item ? maxCapacity)
    ?? ? ParentId se actualiza en BD

Crear Ship
    ?? ? ShipId no puede estar vacío
    ?? ? MaxCapacity > 0
    ?? ? MinCapacity ? 0
    ?? ? MinCapacity ? MaxCapacity

Cargar Item en Ship
    ?? ? Item existe en BD
    ?? ? Peso total cargo no excede MaxCapacity
    ?? ? (Futuro) Validar que Item no está en otro Ship
```

## Decisiones de Diseńo

| Decisión | Razón | Alternativa Considerada |
|----------|-------|-------------------------|
| **Table-Per-Hierarchy (TPH)** | Permite jerarquía polimórfica en una tabla. Consultas simples. | Table-Per-Type (TPT): más tablas pero relaciones complejas |
| **CompositeStorage abstracta** | Fuerza implementación de MaxCapacity en subclases | Usar interfaz: menos control sobre propiedades compartidas |
| **Relación recursiva (ParentId)** | Permite jerarquías profundas. DeleteBehavior.Cascade | Tabla de relación explícita: más complejidad |
| **Repository genérico** | Reutilización de código. GetByTypeAsync<T>() | Repositorio específico por tipo: más código pero más control |
| **DTOs separados** | Desacoplamiento API/BD. Flexibilidad en cambios | Retornar entidades directamente: menos seguro |
| **Ship sin persistencia de Cargo** | Simplificación v1. Ship es cliente del composite | Tabla ShipCargo explícita: posible mejora futura |

- Índice en ParentId para búsquedas de hijos
